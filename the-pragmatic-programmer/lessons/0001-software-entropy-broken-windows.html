Software Entropy & Broken Windows

Why good code rots — and the one habit that stops it

The Research Behind the Metaphor

In the 1980s, criminologists James Wilson and George Kelling published the Broken Windows Theory: a single broken window in a building, left unrepaired, signals that no one cares. This invites more vandalism. The building deteriorates rapidly — not because of any single bad actor, but because of a social norm that disorder is acceptable here.

Thomas and Hunt applied this directly to codebases. One piece of bad code — a messy function, an ignored TODO, a quick hack that "we'll fix later" — signals to every developer who reads it: this codebase accepts disorder. The entropy accelerates.

The cause of software rot is not time or complexity.
It is neglect — the toleration of broken windows.

What Counts as a Broken Window

The book is specific: bad designs, wrong decisions, and poor code. But in practice, a broken window is anything that signals we don't care here:

Broken window
What it signals
Function that does 5 different things, named handleData()
Naming doesn't matter here. Add more unclear names.
TODO comment from 2 years ago: // fix this later
Technical debt is acceptable. Add more.
Test that's been skip-ped for 6 months
Tests don't need to pass. The build being green is optional.
Inconsistent error handling across the module
No convention exists here. Do whatever.

Why This Is Hard to Apply

The broken windows trap is not "I'm lazy." It's "this is a small thing and I'm busy." That reasoning feels rational every single time. The book's counter-argument is that the cumulative effect of each small tolerance is catastrophic — and that once entropy takes hold, reversing it requires heroic effort.

Tip 4 — The Pragmatic Programmer "Don't live with broken windows. Fix each one as soon as it is discovered. If there is insufficient time to fix it properly, then board it up. Perhaps you can comment out the offending code, or display a 'Not Implemented' message, or substitute dummy data instead. Take some action to prevent further damage and to show that you're on top of the situation."

"Board it up" is the key nuance: you don't always have time to fix immediately, but you must acknowledge the window is broken. A comment // FIXME: this logic is wrong, ticket #1234 is a boarded window. An unflagged inconsistency is a broken one.

The Senior Developer Connection

Junior and mid-level developers often see code quality as a personal aesthetic preference ("I like clean code"). The broken windows mental model reframes it as a team dynamics and system health problem: tolerating disorder changes what the team believes is acceptable, which changes what the codebase becomes over 12 months.

Senior engineers hold the line on broken windows not because they're perfectionist, but because they understand entropy. They know that the one messy PR they approve without comment implicitly says "this level is fine here."
