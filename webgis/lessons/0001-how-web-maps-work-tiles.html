How Web Maps Work:
The Tile System

The Core Mental Model

Every web map you have ever seen — Google Maps, OpenStreetMap, satellite views — is built from the same simple idea: the world is cut into a grid of square images called tiles, and your browser assembles them like a puzzle.

This is true for satellite imagery too. When you look at a satellite map, you are not looking at one giant photo. You are looking at thousands of small image tiles stitched together in real time.

Retain this A web map is a grid of square images. Each image is called a tile. The library (MapLibre, Leaflet, etc.) fetches and assembles them automatically. Your job as a front-end developer is to tell the library where to get the tiles and what to draw on top of them.

Zoom Levels

The tile system has a third dimension: zoom level. At zoom 0, the entire world fits in a single tile (256×256 pixels). Every time you zoom in one level, each tile splits into four smaller tiles — so the number of tiles quadruples.

The formula: at zoom level z, there are 4z tiles covering the world. Satellite maps typically go up to zoom 19 or 20. Each additional zoom level doubles the apparent resolution.

XYZ Coordinates

In practice, you will almost never calculate tile coordinates yourself. But understanding XYZ is essential because tile servers are addressed using a URL template that contains these coordinates.

The Tile URL Template

Tile servers expose tiles via a URL pattern. When you configure a map library like MapLibre, you give it this URL template and it fills in the coordinates automatically as the user pans and zooms.

https://example-tiles.com/satellite/{z}/{x}/{y}.jpg

For example, the tile highlighted in yellow above would be fetched from:

https://example-tiles.com/satellite/2/2/1.jpg

When working with satellite imagery at your job, you will likely receive a URL template like this from your tile provider (Mapbox, MapTiler, a custom server). You pass it into MapLibre as a raster source — more on that in a later lesson.

Why this matters for your job When you add satellite imagery in MapLibre, you are not loading one image. You are registering a tile URL template. The library handles all the math: which tiles are visible at the current zoom and pan position, fetching them in parallel, caching them, and stitching the grid together. Your code is just a config object.

Raster vs Vector Tiles

There are two kinds of tiles you'll encounter:

Raster tiles are pre-rendered images (PNG or JPG). Satellite imagery is always raster — it is literally a photograph. The tile arrives as pixels, and the browser just displays it.

Vector tiles contain raw geometry data (roads, boundaries, building outlines) encoded as binary. The browser decodes them and renders them using the library's own style rules. Most modern base maps use vector tiles because they are smaller, faster, and can be restyled. MapLibre is built primarily around vector tiles.

In a typical satellite app, you will use both: a raster source for the satellite photos, and vector sources for the data layers you draw on top (annotations, field boundaries, analysis results).
