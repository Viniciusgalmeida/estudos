The Main Sequence

Lesson 6 · Fundamentals of Software Architecture · Ch. 3 – Modularity

The One Thing to Retain

Core principle Stable modules (many things depend on them) must be abstract. Unstable modules (few things depend on them) can be concrete. Violate this balance and you end up in the Zone of Pain — locked into a concrete module you cannot change — or the Zone of Uselessness — beautiful abstractions nobody uses.

This is the insight that ties the whole modularity chapter together. Cohesion (from Lesson 5) tells you how well a module's parts belong together. Coupling tells you how modules relate to each other. The Main Sequence tells you whether those relationships are structured in a way that can survive change.

Building Block 1 — Coupling Direction

Two types of coupling measure a module's place in the dependency graph:

TermAbbrMeaningIntuition
Afferent coupling Ca How many modules depend on this one Incoming arrows into this module
Efferent coupling Ce How many modules this one depends on Outgoing arrows from this module

Memory hook: Afferent = Arriving (incoming). Efferent = Exiting (outgoing).

Building Block 2 — Instability (I)

Instability I = Ce / (Ce + Ca)    range: 0–1

I = 0 (stable): many things depend on this module, nothing it depends on. A change here breaks callers. You're locked in.
I = 1 (unstable): this module depends on many things, but nothing depends on it. Free to change anytime. A leaf node.

Example: a formatDate utility used by 12 other modules has Ca=12, Ce=0 → I = 0/(0+12) = 0 (maximally stable). A new feature module that imports 5 libraries and is imported by nobody has Ce=5, Ca=0 → I = 5/5 = 1 (maximally unstable).

Building Block 3 — Abstractness (A)

Abstractness A = abstract elements / total elements    range: 0–1

In typed languages: abstract classes and interfaces count as abstract elements. In JavaScript (no classes required): abstractness shows up as dependency injection — accepting a function as a parameter instead of calling a concrete implementation directly.

A concrete module has one specific implementation hardcoded. An abstract module accepts its behavior from the outside — callers can swap in different implementations without touching the module itself.

The Main Sequence

Plot Abstractness (A) on the Y axis and Instability (I) on the X axis. Every module in your codebase lands somewhere on this grid.

The diagonal line is the Main Sequence — the ideal trade-off. Modules should live near it. The further a module drifts from the line, the more problematic it is.

Zone of Pain

Zone of Pain — stable + concrete Many modules depend on it (I ≈ 0), but it has no abstraction (A ≈ 0). It is impossible to change without breaking callers, and it offers no way to swap behavior.

Classic Zone of Pain in JavaScript: a utility module used everywhere, with a concrete implementation baked in:

// notifications.js — 12 other modules import this directly
import twilioClient from 'twilio';
import { getUserPhone } from './users';

function sendNotification(userId, message) {
  const phone = getUserPhone(userId);
  return twilioClient.messages.create({ to: phone, body: message });
}

export { sendNotification };

Ca = 12 (12 modules import it), Ce = 2 (imports twilio + users) → I = 2/14 ≈ 0.14 (stable). A = 0 (nothing abstract — one hardcoded Twilio implementation). Distance from sequence = |0 + 0.14 − 1| = 0.86. Deep in the Zone of Pain.

The problem isn't that it sends SMS. The problem is that when the product team says "add email notifications" or "switch from Twilio to AWS SNS," you must touch this module and all 12 callers are at risk.

Zone of Uselessness

Zone of Uselessness — abstract + unstable Lots of abstraction (A ≈ 1), but nothing depends on it (I ≈ 1). Beautiful design that nobody uses. 
// notificationStrategy.js — abstract, but nothing imports it
function createNotificationSender(sendFn) {
  return function(userId, message, contact) {
    return sendFn(contact, message);
  };
}

// Fully pluggable. Zero callers. The abstraction goes unused.
export { createNotificationSender };

Ca = 0 (nobody imports it), Ce = 0 → I = 0/0, treated as 1 (unstable). A ≈ 1 (fully abstract). Distance = |1 + 1 − 1| = 1. Equally far from the Main Sequence, in the opposite direction. The design effort was wasted.

On the Main Sequence

The fix for the Zone of Pain example: inject the send behavior, so callers retain stable access to the contract while the implementation can change freely. 

// notifications.js — stable contract, abstract implementation
function sendNotification(userId, message, send) {
  const contact = getContact(userId);
  return send(contact, message);   // caller provides the "how"
}

export { sendNotification };

// ─── callers decide the channel ───────────────────────────────

// In one context: SMS
sendNotification(id, msg, (contact, text) =>
  twilioClient.messages.create({ to: contact.phone, body: text })
);

// In another: email
sendNotification(id, msg, (contact, text) =>
  emailClient.send({ to: contact.email, subject: 'Alert', text })
);

// In tests: no network calls
sendNotification(id, msg, (contact, text) => {
  captured.push({ contact, text });
});

Now the module is still stable (12 callers) and abstract (accepts the implementation). Moving from the Zone of Pain onto the Main Sequence didn't require a big refactor — just invert one dependency.

Distance from the Main Sequence

Distance D = |A + I − 1|    range: 0–1    (0 = ideal, 1 = worst)

D is the key metric. Tools like Structure101 or static analysis plugins can compute it across an entire codebase, producing a scatter plot. Modules far from D=0 are architectural risk zones.

Practically: when you see a module with many callers, ask yourself whether it's abstract enough to accommodate the changes it will be asked to support. That question is D in plain English.
