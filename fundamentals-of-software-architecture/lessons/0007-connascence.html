Connascence

Lesson 7 · Fundamentals of Software Architecture · Chapter 3

The Core Idea

Coupling metrics (Ca, Ce, I) from the last lesson count how many dependencies exist. Connascence names what kind of dependency it is — and that distinction drives refactoring decisions.

Definition Two components are connascent if a change in one requires a corresponding change in the other to keep the system correct. The stronger the connascence, the harder that change becomes.

Connascence was introduced by Meilir Page-Jones and later refined in the software architecture community. Its value is vocabulary: instead of "these two things are coupled," you can say "these have Connascence of Position" — and immediately know the severity and the fix.

The Strength Ladder

Connascence types are ordered by strength — how difficult it is to discover and refactor the coupling. Weaker connascence is always preferable, especially across module boundaries.

Key rule Dynamic connascence is always harder to reason about than static — bugs are invisible until runtime. When you spot dynamic connascence, ask whether it can be converted to a static form.

Three Refactoring Moves in Code

The practical skill is recognizing a connascence type and knowing the move to weaken it.

Move 1: CoM → CoN — Replace magic values with named constants.

// CoM — two files must silently agree that 1 means "active"
function activateUser(user) {
  return { ...user, status: 1 };         // what is 1?
}

function isActive(user) {
  return user.status === 1;               // same magic number, different file
}

// ─────────────────────────────────────────────
// CoN — both files now depend only on the name
const USER_STATUS = { ACTIVE: 'active', INACTIVE: 'inactive' };

function activateUser(user) {
  return { ...user, status: USER_STATUS.ACTIVE };
}

function isActive(user) {
  return user.status === USER_STATUS.ACTIVE;
}

Renaming ACTIVE is a mechanical find-and-replace. Hunting every 1 in the codebase is not.

Move 2: CoP → CoN — Replace positional arguments with a named-object parameter.

// CoP — callers must know: name first, email second, role third
function createUser(name, email, role) {
  return { name, email, role };
}

createUser('Alice', 'alice@co.com', 'admin');
createUser('alice@co.com', 'Alice', 'admin'); // silent bug — no error thrown

// ─────────────────────────────────────────────
// CoN — order no longer matters
function createUser({ name, email, role }) {
  return { name, email, role };
}

createUser({ name: 'Alice', email: 'alice@co.com', role: 'admin' });

Move 3: CoA — the hardest to fix. Two modules implement the same algorithm independently. If the algorithm changes, both must be updated in sync.

// CoA — both functions re-implement the same encoding
function encodeToken(userId) {
  return btoa(`user:` + userId);           // base64 of "user:123"
}

function decodeToken(token) {
  const raw = atob(token);
  return raw.split(':')[1];               // must know the "user:" prefix
}

// ─────────────────────────────────────────────
// CoN — algorithm lives once; consumers depend only on name
function encodeToken(userId) {
  return btoa(`user:` + userId);
}

function decodeToken(token) {
  return encodeToken;                      // wrong — see below
}

// Actually: extract the shared rule into one function both call
function tokenPrefix() { return 'user:'; }

function encodeToken(userId) {
  return btoa(tokenPrefix() + userId);
}

function decodeToken(token) {
  const raw = atob(token);
  return raw.replace(tokenPrefix(), '');  // single source of truth
}

Locality Changes Everything

Connascence within a single module is acceptable at higher strengths. Connascence across module or service boundaries must be minimized. The farther apart two components are, the weaker their connascence should be.

Distance Acceptable strength Reasoning
Same function Any Change is local, one place to update
Same module/file Up to CoA Grep finds it; refactor is bounded
Across modules CoN, CoT only Renames tool-supported; meaning/position drift silently
Across services CoN only Runtime-only coupling (CoE, CoV, CoI) becomes distributed system bug

How This Extends What You Already Know

Ca/Ce coupling metrics from last lesson tell you how many things depend on a module. Connascence tells you how deeply they depend on it. A module can have low Ce but strong CoA — only one outgoing dependency, but it requires synchronizing a shared algorithm. That's still dangerous.

Use both lenses together: Ca/Ce for quantity, connascence for quality.

Check your understanding

1. A payment module and a reporting module both contain this check: if (type === 2) — both silently agree that 2 means "refund". What type of connascence is this?

2. You refactor send(to, subject, body) into send({ to, subject, body }) across callers. What connascence type did you eliminate?

3. A service A must call initCache() before calling getUser(), or the app crashes at runtime. There is nothing in source code that enforces this order. What connascence type is this?

4. Dynamic connascence is generally considered worse than static connascence. Why?

5. You are reviewing an API that spans two separate services. Which connascence strength is safest to allow across that boundary?
