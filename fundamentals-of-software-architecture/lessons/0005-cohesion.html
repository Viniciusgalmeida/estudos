Cohesion

Lesson 5 · Fundamentals of Software Architecture · Ch. 3 – Modularity

The One Thing to Retain

Core principle Cohesion measures how much the parts of a module belong together. High cohesion = every part serves one purpose. Low cohesion = a module that has crept into multiple responsibilities and will resist change.

The book's simplest definition: "the degree to which the parts of a module belong together." If you can honestly say "this class/module does X and Y and Z," that's a warning signal — it probably lacks cohesion.

Why architects care: low-cohesion modules become change magnets. Two unrelated business rules live in the same place → every small change risks touching unrelated logic → higher coupling, harder testing, slower delivery.

The Seven Types (Worst → Best)

The book lists seven cohesion types. The ones you'll encounter in real code:

Coincidental
worst
Logical
Temporal
Procedural
Communicational
Sequential
Functional
best

Coincidental — parts are grouped by accident (e.g., a Utilities class that has no real theme).
Logical — parts perform similar operations on different types (InputValidator.validateEmail / validatePhone / validateAge), but the caller must pass a flag to pick behavior.
Temporal — parts must run at the same time (SystemInit.run() does DB connect + logging setup + cache warm).
Communicational — parts operate on the same data but do different things (OrderProcessor that both calculates tax and sends confirmation email).
Functional — everything serves a single, well-defined purpose (TaxCalculator). Best.

LCOM — Making Cohesion Measurable

The book introduces LCOM (Lack of Cohesion in Methods) as an objective metric. High LCOM = low cohesion (bad). Low LCOM = high cohesion (good).

The intuition: imagine you can see which instance variables each method touches. If methods cluster into independent groups — one group touches fields A+B, another touches fields C+D — LCOM is high. The class is secretly two classes sharing a file. 

// High LCOM — two disconnected clusters:
class Utilities {
  private name: string;
  private email: string;

  formatName() { return this.name.trim(); }      // touches: name
  validateEmail() { return this.email.includes('@'); } // touches: email
  // ↑ name and email never appear in the SAME method → LCOM high
}

// Low LCOM — methods share fields:
class User {
  private name: string;
  private email: string;

  describe() { return `${this.name} <${this.email}>`; }  // touches: name + email
  update(n, e) { this.name = n; this.email = e; }           // touches: name + email
}

LCOM is a signal, not a sentence. A high score means "look here" — it doesn't automatically mean "split this." Sometimes a utility class is intentional. But if you see LCOM spiking during static analysis, it's the tool telling you: this module is doing more than one thing.

Why This Matters at the Architecture Level

Cohesion isn't just a class-design concern. It scales up. A service that handles user authentication and billing notifications has low cohesion at the service level. A team owning both the payment domain and the reporting domain will feel the same friction — competing priorities, tangled releases.

The architect's reflex Whenever you see a module (class, service, team) described with "and," ask: do these two things change together, or independently? If independently → split. If together → keep together.

Connection to Prior Lessons

Cohesion is the first half of modularity. Next comes coupling — how dependent modules are on each other. The goal is always: high cohesion, low coupling. See also: Lesson 3 — Laws of Software Architecture (trade-offs apply here too: splitting for cohesion increases the number of components you must coordinate).
