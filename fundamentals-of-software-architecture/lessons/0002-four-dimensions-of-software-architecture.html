The Four Dimensions of Software Architecture

Fundamentals of Software Architecture — Richards & Ford, 2020

Why this is the most important concept from pp. 1–8: Every other chapter in this book adds detail to one of these four dimensions. If you retain only one framework from Ch. 1, it's this one — it gives you a vocabulary for every design conversation you'll ever have.
The single biggest mistake developers make when discussing architecture is confusing structure for the whole picture. Saying "we use microservices" describes one dimension. Software architecture is defined by four dimensions working together.

The Four Dimensions

Dimension 1 Structure

The type of architecture style implemented. This is what most people mean when they say "the architecture" — but it's only one piece.

microserviceslayeredpipelinemicrokernelevent-driven
Dimension 2 Architecture Characteristics

The success criteria of the system — the "-ilities." These define what the system must do well, independent of what it does. Two systems can share the same structure but have radically different characteristics.

availabilityscalabilityperformancesecuritytestabilitydeployabilityfault tolerance
Dimension 3 Architecture Decisions

The rules for how the system must be constructed. These are constraints — explicit, binding, and enforceable. Breaking them requires an explicit decision to do so.

"only the service layer may access the DB""no cross-service DB joins"
Dimension 4 Design Principles

Guidelines, not rules. Preferred approaches that teams should follow but may deviate from with good reason. They guide without constraining.

"prefer async messaging for inter-service calls""avoid shared libraries where possible"

The Critical Distinction: Decisions vs. Principles

These two dimensions trip people up most often. The difference is binding force:

Architecture Decision (Rule)

"Services must not share a database." Violation requires an architecture change request. Non-negotiable.

Design Principle (Guideline)

"Prefer async messaging for performance." A team may use sync HTTP with justification. Negotiable.

Recall Check

No peeking. Retrieve from memory. These questions test application, not just label recognition.

1. Your team says: "Our architecture is microservices." Which single dimension have they described?

Correct. "Microservices" is a structure — Dimension 1. It tells you the style, but nothing about the -ilities, the rules, or the guidelines. The common mistake is treating this single dimension as if it describes the full architecture.

2. A new system must handle 100k concurrent users and recover from failures within 30 seconds. Which dimension captures these requirements?

Correct. Scalability and recoverability are architecture characteristics — the -ilities. Dimension 2 defines success criteria independent of the system's domain logic.

3. "No service may directly call another service's database." This is an example of:

Correct. It's phrased as a hard constraint ("may not"), so it's Dimension 3 — an architecture decision. If it were a design principle, it would say "avoid" or "prefer not to," and a team could deviate with justification.

4. "Prefer asynchronous communication between services when latency permits." This is:

Correct. "Prefer" signals a guideline, not a rule. Dimension 4 — design principles — allows deviation. The team can use sync HTTP if there's a valid reason.

5. Two systems both use a layered architecture. One needs 99.99% uptime, the other needs 99%. Which dimension makes them architecturally different even though their structure is the same?

Correct. This is the key insight: same structure, different architectures. Availability (an architecture characteristic — Dimension 2) changes everything about how redundancy, failover, and deployment are designed, even inside the same structural style.
